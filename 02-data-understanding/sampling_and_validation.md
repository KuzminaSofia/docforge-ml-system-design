# Алгоритм формирования выборки и стратегия валидации

## 1. Исходные данные

DocLayNet поставляется с **предопределёнными splits** от авторов:

| Split | Страниц | Доля |
|---|---|---|
| train | 69 375 | 85.8% |
| validation | 6 489 | 8.0% |
| test | 4 999 | 6.2% |
| **Всего** | **80 863** | **100%** |

Split сделан **на уровне документов, не страниц**: страницы одного исходного PDF целиком уходят в один сплит. Это критично — иначе разные страницы одного отчёта попали бы в train и test, что дало бы data leakage (модель видела бы похожие лейауты).

**Решение: используем официальный split без изменений.** Он сбалансирован по доменам и защищён от leakage.

## 2. Формирование выборки для обучения

### 2.1. Базовый сценарий — full corpus training

Train (69 375 страниц) используется целиком. Это даёт максимальное разнообразие лейаутов.

### 2.2. Альтернатива — domain-targeted subset

Целевая аудитория DocForge — техническая документация. Для production-fine-tune можем использовать subset:

```
subset = train.filter(lambda x: x["doc_category"] in {"manuals", "patents", "laws_and_regulations"})
```

Размер ≈ 26 000 страниц. Преимущество — быстрее обучение, ближе к продакшен-распределению. Риск — модель хуже обобщается на «нестандартные» лейауты (финансовые отчёты, научные статьи), что может встретиться у клиента.

**Решение:** обучаем **две модели** — full и domain-targeted — и сравниваем на validation. Выбираем ту, что выше на manuals.

### 2.3. Балансировка классов

Дисбаланс 100× между Text и Title требует mitigation. Варианты:

| Подход | Pros | Cons |
|---|---|---|
| Class-balanced sampling (oversample rare) | простой, работает | мульти-аугментация одних и тех же примеров → overfit |
| Focal loss (γ=2.0) | без изменения сэмплинга, проверенный SOTA | требует подбора γ |
| Repeat factor sampling (RFS) | хорошо для object detection | усложняет батчинг |

**Решение:** focal loss. Это стандарт для COCO-style детекции (RetinaNet, ATSS) и не требует менять loader.

### 2.4. Аугментации

| Аугментация | Зачем | Параметры |
|---|---|---|
| Random resize | устойчивость к разрешению | scale 0.5–1.5 |
| Random crop | устойчивость к обрезке | min IoU 0.5 |
| Color jitter | сканы разного качества | brightness ±0.3, contrast ±0.3 |
| Gaussian blur | низкокачественные PDF | sigma 0–2 |
| JPEG compression | эмуляция сканов | quality 50–95 |
| Horizontal flip | **НЕ применяем** | flip ломает порядок чтения (LTR) |
| Rotation ±2° | сканы под лёгким наклоном | вероятность 0.3 |

## 3. Стратегия валидации

### 3.1. Уровень 1 — Validation set (in-loop)

После каждой epoch — оценка на validation (6 489 страниц).

Что считаем:
- **Overall mAP@0.5:0.95** — основной KPI
- **Per-class AP** для всех 11 классов
- **Per-domain mAP** — особенно manuals
- **Loss** (cls / bbox / objectness)

Early stopping: patience 5 epoch по overall mAP.

### 3.2. Уровень 2 — Test set (финальная оценка)

Test set (4 999 страниц) трогаем **один раз** перед коммитом модели в продакшен. Не используем для tuning гиперпараметров — иначе test перестаёт быть честным held-out.

Что считаем дополнительно:
- **Per-class AP с доверительными интервалами** (bootstrap 1000 ресэмплов)
- **Error analysis** — ручной разбор 50 худших примеров по mAP (где модель сильнее всего проседает)

### 3.3. Уровень 3 — Internal RU validation (production-fit)

DocLayNet — латиничный. DocForge продаётся в РФ. Нужен **внутренний RU-validation set**.

Сборка:
- 300–500 страниц русских PDF: ГОСТы, тех. регламенты, мануалы, паспорта оборудования
- Двойная разметка по той же 11-классной схеме
- IoU 0.7 как порог согласия аннотаторов
- Конфликты → третий эксперт (arbitration)

Этот сет — финальная gate перед релизом: если на нём mAP падает > 10 п.п. относительно DocLayNet test → требуется domain adaptation / fine-tune на RU подвыборке.

### 3.4. Уровень 4 — Production shadow mode

Первые 2 недели после релиза — модель работает в shadow: предсказывает, но результат сравнивается с текущим Docling backend. Расхождения логируются для post-release анализа.

## 4. Защита от типичных ошибок

| Ошибка | Как защищаемся |
|---|---|
| Leakage между train и test | использован document-level split от авторов |
| Cherry-picking гиперпараметров на test | test трогаем один раз; tuning — только на val |
| Smoothed mAP скрывает деградации редких классов | в репортах — обязательно per-class AP |
| Overfit на доминирующий домен (financial) | per-domain mAP в репорте, ранний стопаем по manuals |
| Дрифт после релиза | shadow mode + еженедельный re-evaluation на новых документах клиента |

## 5. Метрики приёмки модели

Модель допускается в production, если **все** условия выполнены на test set DocLayNet:

| Метрика | Порог |
|---|---|
| Overall mAP@0.5:0.95 | ≥ 0.70 |
| Per-class AP — Title | ≥ 0.55 |
| Per-class AP — Table | ≥ 0.75 |
| Per-class AP — Section-header | ≥ 0.65 |
| mAP на manuals | ≥ 0.72 |

**Дополнительно** на internal RU validation:

| Метрика | Порог |
|---|---|
| Overall mAP@0.5:0.95 | ≥ 0.60 |
| F1 по Text + Section-header | ≥ 0.75 |

Если хотя бы один порог не пройден — модель не релизим, идём в error analysis и доcборку данных.

## 6. Cross-validation: нужна или нет?

Для DocLayNet — **нет**. Корпус 80k страниц + фиксированные splits + дисбаланс по документам — k-fold не даст более точной оценки, чем единичный test, а вычислительно дороже в k раз.

Для **internal RU set** (300–500 страниц) — **да**, **stratified 5-fold по доменам**. Маленький объём → одна выборка test даст шумную оценку.
