# Business Understanding

# DocForge — платформа подготовки знаний для корпоративных RAG-систем

## Описание проекта

DocForge — интеллектуальная платформа подготовки, проверки и загрузки технической документации в корпоративные базы знаний и RAG-системы.

Платформа предназначена для автоматизации полного цикла подготовки документов перед их использованием в системах поиска знаний, Retrieval-Augmented Generation (RAG), корпоративных чат-ботах и цифровых ассистентах.

DocForge позволяет загружать документы, выполнять их автоматическую обработку, контролировать качество результатов, редактировать ошибки, подготавливать данные для векторных баз знаний и управлять процессом наполнения корпоративной базы знаний через удобный пользовательский интерфейс.

Основная идея проекта заключается в том, чтобы передать процесс подготовки знаний предметным экспертам и снизить зависимость от специалистов по данным при сопровождении корпоративных систем искусственного интеллекта.

# 2. Бизнес-цель проекта

Во многих компаниях внедряются корпоративные AI-ассистенты и RAG-системы, работающие с внутренней документацией.

Основной проблемой становится не создание самого ассистента, а постоянное поддержание базы знаний в актуальном состоянии.

Для добавления нового документа в базу знаний обычно необходимо:

* обработать документ;
* проверить качество извлечения информации;
* выполнить очистку данных;
* разбить документ на чанки;
* загрузить данные в векторную базу знаний;
* проверить качество индексации.

Данные операции часто выполняются специалистами по данным и разработчиками, что приводит к высоким затратам времени и замедляет обновление базы знаний.

Бизнес-цель проекта — сократить затраты на сопровождение корпоративной базы знаний и передать процесс подготовки документов предметным экспертам без необходимости взаимодействия с кодом и ML-инфраструктурой.

# 3. Существующие решения и их ограничения

В компании уже существуют технологии обработки технической документации.

Для обработки документов используются специализированные пайплайны, которые позволяют:

* выполнять OCR;
* извлекать текст из PDF и сканов;
* восстанавливать структуру документа;
* извлекать таблицы;
* обрабатывать документы низкого качества;
* преобразовывать документы в структурированный формат.

Особенностью технической документации является сложность её обработки.

Документы могут содержать:

* некачественные сканы;
* фотографии страниц;
* таблицы сложной структуры;
* схемы и чертежи;
* смешанные языки;
* различные шаблоны оформления.

Поэтому для качественной обработки недостаточно использовать один инструмент или одну модель. Обычно применяется набор специализированных пайплайнов и алгоритмов обработки.

Несмотря на наличие технологий обработки документов, их использование требует участия специалистов по данным и разработчиков, поскольку работа выполняется через код и внутренние сервисы.

Предметные эксперты не имеют удобного инструмента для самостоятельной подготовки документов.

DocForge не заменяет существующие технологии обработки документов, а предоставляет единый интерфейс управления, проверки качества и подготовки данных для последующей загрузки в корпоративную базу знаний.

Именно они смогут оценивать качество обработки документов и качество разбиения документов на смысловые фрагменты.

# 5. Риски проекта

## Риск 1. Низкое качество исходных документов

Документы могут содержать сканы плохого качества, фотографии страниц или повреждённые фрагменты.

Меры снижения риска:

* использование OCR;
* ручная проверка результатов;
* возможность редактирования результатов обработки.

## Риск 2. Высокое разнообразие форматов документов

Документы могут существенно отличаться по структуре.

Меры снижения риска:

* модульная архитектура обработки;
* возможность подключения различных пайплайнов обработки.

## Риск 3. Недостаточное качество автоматического разбиения на чанки

Ошибки разбиения могут ухудшать качество поиска в RAG-системах.

Меры снижения риска:

* визуальная проверка чанков;
* возможность ручного редактирования.

## Риск 4. Низкая вовлечённость пользователей

Пользователи могут предпочесть существующие процессы работы.

Меры снижения риска:

* простой интерфейс;
* минимизация ручных операций;
* интеграция с существующими процессами компании.

# 6. Решаемые задачи с точки зрения аналитики (Data Mining Goals)

## Основная задача

Автоматическое структурирование технических документов.

Вход:

* PDF;
* DOCX;
* DOC.

Выход:

* структурированное представление документа;
* метаданные документа;
* выделенные логические разделы документа.

## Дополнительные задачи

* извлечение сущностей из технической документации;
* классификация документов;
* семантическое разбиение документов на чанки;
* подготовка данных для использования в RAG-системах;
* оценка качества подготовленных данных перед загрузкой в базу знаний.

# 7. Метрики качества

На текущем этапе проекта предполагается использование различных метрик качества в зависимости от решаемой аналитической задачи.

Финальный набор метрик будет уточнен после анализа данных на этапе Data Understanding.

## Извлечение сущностей

Предварительно планируется использовать следующие метрики:

- Precision;
- Recall;
- F1-score.

## Классификация документов

Предварительно планируется использовать следующие метрики:

- Accuracy;
- Precision;
- Recall;
- F1-score.

## Семантическое разбиение документов

Для оценки качества семантического разбиения документов на чанки планируется использовать экспертную оценку.

Основные критерии оценки:

- смысловая целостность чанков;
- отсутствие потери контекста;
- пригодность чанков для использования в RAG-системах.

# 8. Критерии успешности проекта

Проект считается успешным при выполнении следующих условий.

## Бизнесовые критерии

* снижение времени подготовки документов для базы знаний не менее чем на 50%;
* снижение участия команды Data Science в процессе наполнения базы знаний;
* сокращение времени публикации нового документа в базе знаний.

## Пользовательские критерии

* пользователь способен самостоятельно подготовить документ к загрузке;
* пользователь способен проверить результаты обработки без участия разработчиков;
* не менее 80% автоматически сформированных чанков принимаются экспертами без существенных исправлений.

# 9. План проекта

Проект реализуется в соответствии с методологией CRISP-DM.

| Этап                   | Результат                                                    |
| ---------------------- | ------------------------------------------------------------ |
| Business Understanding | Формализация проблемы и требований                           |
| Data Understanding     | Анализ типов документов и источников данных                  |
| Data Preparation       | Подготовка документов и формирование датасетов               |
| Modeling               | Разработка методов структурирования и чанкинга               |
| Evaluation             | Оценка качества обработки документов                         |
| Deployment             | Развёртывание платформы DocForge и интеграция с базой знаний |

# 10. Ожидаемый результат

Результатом проекта должна стать платформа, позволяющая предметным экспертам самостоятельно подготавливать техническую документацию для корпоративных баз знаний и RAG-систем без необходимости взаимодействия с кодом, ML-пайплайнами и внутренней инфраструктурой компании.

DocForge должен стать связующим звеном между технологиями обработки документов и конечными пользователями, обеспечивая удобный процесс сопровождения корпоративной базы знаний.
