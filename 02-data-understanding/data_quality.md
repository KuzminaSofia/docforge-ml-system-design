# Качество разметки DocLayNet и предложения по улучшению

## 1. Источник оценки

DocLayNet содержит **redundantly annotated subset**: часть страниц размечена несколькими экспертами независимо. Это даёт **прямую измеримую оценку качества разметки** через inter-annotator agreement (IAA).

- Double-annotated: **7 059 страниц** (~ 8.7% корпуса)
- Triple-annotated: **1 374 страницы** (~ 1.7% корпуса)
- Метрика согласованности: **mAP@0.5:0.95** между парами аннотаторов (см. оригинальную статью, Table 3)

## 2. Результаты по классам

| Класс | Human mAP, % | Оценка качества | Причина расхождений |
|---|---|---|---|
| Formula | 100 | отличное | чёткие визуальные признаки (LaTeX-блоки) |
| Page-footer | 89 | хорошее | стабильное положение и шрифт |
| List-item | 88 | хорошее | bullet-маркеры, отступы |
| Page-header | 86 | хорошее | стабильное положение |
| Caption | 84 | приемлемое | граница между caption и body неоднозначна |
| Text | 84 | приемлемое | склейка соседних абзацев — частый спор |
| Footnote | 83 | приемлемое | граница между footnote и page-footer |
| Table | 82 | приемлемое | границы крупных таблиц с подписями |
| Section-header | 79 | посредственное | где заканчивается Title и начинается Section-header |
| Picture | 71 | слабое | сложные сцены, диаграммы vs формулы |
| **Title** | **60** | **слабое** | пограничный класс с Section-header |

**Средний human mAP по корпусу: ≈ 83%.**

## 3. Интерпретация для DocForge

### 3.1. Существует объективный потолок качества

Никакая ML-модель не сможет получить mAP выше ≈ 83%, потому что сами эксперты так не делают. **Это не баг, это естественное свойство задачи** — некоторые границы объективно неоднозначны.

Следствие: целевой mAP DocForge MVP не должен превышать **75–80%**. Цели вида «≥ 95% качества» нереалистичны и приведут к over-fitting на тестовом множестве.

### 3.2. Самые проблемные классы — критичны для продукта

- **Title (60%)** — это **первый чанк** документа в RAG. Ошибки на нём приводят к тому, что в Knowledge Base попадают документы без названия. → высокая бизнес-цена ошибки.
- **Section-header (79%)** — определяет **иерархию чанков**. Ошибка → плохая структура RAG-контекста.
- **Picture (71%)** — без правильной локализации картинок в чанки попадает мусор.

### 3.3. Слабая корреляция с классовым дисбалансом

Title (5k bbox, 60% mAP) — мало данных и плохая согласованность.
Formula (25k bbox, 100% mAP) — мало данных, но идеальная согласованность.

→ Качество разметки слабее зависит от объёма данных, чем от объективной сложности границ.

## 4. Предложения по повышению качества разметки

Если бы нам нужно было собирать **внутреннюю русскоязычную разметку для DocForge** (вместо/в дополнение к DocLayNet), мы бы заложили:

### 4.1. Уточнённая labeling guideline

Оригинальный guide DocLayNet — общий. Для нашего домена (RU тех. документация) нужно явно прописать пограничные случаи:

- **Title vs Section-header**: «Title — это название всего документа, обычно на первой странице, шрифт самый крупный. Section-header — название главы/параграфа.» + примеры.
- **Caption vs Text** под рисунком: «Caption начинается с маркера "Рис. N", "Figure N" или "Таблица N".» + примеры.
- **Picture vs Formula**: «Formula — содержит математические символы и подается в одну строку или блок; Picture — растровое или векторное изображение, диаграмма, схема.»
- **Footnote vs Page-footer**: «Footnote — текст под чертой внизу страницы, относится к содержательной части. Page-footer — номер страницы, копирайт, не зависит от содержания.»

### 4.2. Cross-annotation на «горячих» классах

Title, Section-header, Picture, Caption — **обязательная двойная разметка** + arbitration третьим экспертом при расхождении (IoU < 0.7 или class mismatch).

### 4.3. Active learning loop

Сначала — модель обучается на 1000 страниц. Затем модель показывает аннотаторам страницы с **наибольшей неопределённостью предсказаний** (low confidence + high entropy между классами). Аннотатор размечает именно их → выигрыш в качестве при меньшем объёме разметки.

### 4.4. Валидация консистентности

Перед закрытием новой партии разметки — автопроверки:

- bbox не пересекаются с другими bbox того же класса больше чем на 30% IoU;
- сумма площадей всех bbox на странице ≤ 0.95 × площадь страницы;
- на странице ровно один Title (если есть);
- Caption присутствует только рядом с Picture/Table (расстояние < 100 px).

Страницы, не прошедшие автопроверки, отправляются на ручную верификацию.

### 4.5. Версионирование labeling guideline

Любое изменение guideline → перепроверка ранее размеченного на затронутых классах. Иначе модель учится на «дрифтующей» разметке.

## 5. Что делаем прямо сейчас

DocLayNet не модифицируем — это open-source бенчмарк. Используем как есть, но:

- Берём за метрику **per-class AP**, а не только среднее mAP — чтобы видеть деградации на Title и Picture
- В test-репорт включаем **error analysis по 50 случайным ошибкам на Title** — это даст качественное понимание, где модель не дотягивает до human level
- Готовим внутренний RU-валидационный сет с учётом guideline из п. 4.1 — минимум 300 страниц с двойной разметкой
