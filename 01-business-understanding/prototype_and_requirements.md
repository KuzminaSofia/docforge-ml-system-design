# Prototype and Requirements

# DocForge — прототип продукта

## 1. Ключевая гипотеза

Если предоставить техническим экспертам веб-платформу для самостоятельной подготовки документов к загрузке в корпоративную базу знаний, то не менее 80% документов смогут быть подготовлены без участия команды Data Science в течение одного месяца эксплуатации системы.

Мы ожидаем такой результат, потому что большая часть операций по подготовке документов является типовой и может быть автоматизирована через удобный пользовательский интерфейс.

---

# 2. Метрики прототипа

## Продуктовые метрики

| Метрика | Целевое значение | Способ измерения | Частота |
|----------|----------|----------|----------|
| Доля документов, обработанных без участия Data Science | ≥ 80% | журнал операций системы | еженедельно |
| Среднее время подготовки документа | менее 15 минут | логирование действий пользователя | еженедельно |
| Доля успешно загруженных документов | ≥ 90% | статистика задач | ежедневно |

## Технические метрики

| Метрика | Целевое значение | Способ измерения | Частота |
|----------|----------|----------|----------|
| Успешность обработки документов | ≥ 95% | мониторинг пайплайнов | ежедневно |
| Время обработки документа | менее 5 минут | журнал задач | ежедневно |
| Доля чанков, принятых без ручного редактирования | ≥ 80% | экспертная оценка | еженедельно |

---

# 3. Что входит в MVP

В первую версию прототипа входят:

* загрузка PDF, DOCX и DOC;
* автоматическая обработка документов;
* просмотр структуры документа;
* семантическое разбиение документа на чанки;
* визуальная проверка чанков;
* ручное редактирование чанков;
* загрузка подготовленных данных в базу знаний;
* история обработки документов.

---

# 4. Что намеренно не входит в MVP

Следующие функции не реализуются на этапе прототипа:

* интеграция с корпоративными системами документооборота;
* разграничение доступа по сложной ролевой модели;
* мультимодальная обработка изображений и схем с использованием VLM;
* автоматическое построение графов знаний;
* автоматическое дообучение моделей обработки документов;
* поддержка нескольких корпоративных баз знаний одновременно.

Указанные функции не являются критичными для проверки основной гипотезы и существенно увеличивают сроки разработки MVP.

---

# 5. Используемые готовые компоненты

Для ускорения разработки прототипа используются готовые технологические компоненты и open-source решения.

| Компонент | Решение |
|------------|------------|
| Backend API | FastAPI |
| База данных | PostgreSQL |
| Очередь задач | RabbitMQ |
| Контейнеризация | Docker |
| OCR и обработка документов | Docling, OCR-движки и специализированные библиотеки обработки документов |
| Хранилище данных | S3-совместимое объектное хранилище |
| Векторная база знаний | существующая корпоративная база знаний |

Готовые решения используются только как строительные блоки системы и не определяют основную бизнес-логику продукта.

---

# 6. Что разрабатывается самостоятельно

Ключевой частью проекта является собственное ядро подготовки знаний (Knowledge Preparation Engine), реализуемое в рамках платформы DocForge.

Техническая документация существенно отличается от обычных текстовых документов и может содержать:

* некачественные сканы;
* сложные таблицы;
* схемы и чертежи;
* смешанный текстовый и графический контент;
* различные шаблоны оформления.

Поэтому качественная обработка документа не может быть реализована с использованием одного инструмента или одной модели.

В рамках проекта разрабатывается собственный пайплайн подготовки знаний, который объединяет несколько этапов обработки:

* предварительный анализ документа;
* выбор стратегии обработки;
* OCR и извлечение текста;
* восстановление структуры документа;
* обработка таблиц;
* извлечение метаданных;
* семантическое разбиение документа на чанки;
* проверка качества полученного результата;
* подготовка данных для загрузки в корпоративную базу знаний.

Также самостоятельно разрабатываются:

* пользовательский интерфейс подготовки документов;
* редактор результатов обработки;
* система контроля качества;
* механизм публикации документов в базу знаний;
* инструменты сопровождения корпоративных RAG-систем.

Именно данная логика является основным интеллектуальным компонентом платформы и определяет её ценность для конечных пользователей.

---

# 7. Главный риск прототипа

Главный риск проекта заключается в том, что предметные эксперты могут не принять систему и продолжить использовать существующий процесс взаимодействия с командой Data Science.

Прототип должен показать, что подготовка документа через DocForge требует меньше времени и не требует специальных технических знаний.

---

# 8. Формат прототипа

Прототип представляет собой веб-приложение.

Пользовательский сценарий:

1. Пользователь загружает документ.
2. Система выполняет обработку документа.
3. Пользователь проверяет результат.
4. Пользователь редактирует чанки при необходимости.
5. Пользователь публикует документ в базу знаний.

---

# 9. Критерии остановки теста

Тест считается успешным, если одновременно выполняются следующие условия:

* не менее 80% документов подготовлены без участия Data Science;
* не менее 80% чанков принимаются экспертами без существенных исправлений;
* среднее время подготовки документа не превышает 15 минут.

Тест считается неуспешным, если:

* менее 50% документов могут быть подготовлены пользователями самостоятельно;
* требуется постоянное участие специалистов по данным;
* большинство документов требует существенной ручной переработки.
